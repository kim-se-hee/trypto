## 1. 애그리거트 경계 설정

무엇을 함께 묶나
- 도메인 규칙상 함께 생성되는 구성 요소는 한 애그리거트일 가능성이 높다.
- 함께 변경되는 빈도가 높은 객체는 한 애그리거트일 가능성이 높다.
- "A가 B를 갖는다"는 요구사항이 있다고 A와 B가 한 애그리거트로 묶이는 건 아니다.

경계의 기본 감각
- 대다수 애그리거트는 엔티티 한 개만 갖고, 두 개 이상으로 구성되는 경우는 드물다. 이를 기준점으로 삼으면 경계 설계가 쉬워진다.

## 2. 애그리거트 루트와 일관성

루트의 역할
- 루트가 일관성(비즈니스 규칙이 어긋나지 않게 하는 것)을 책임진다.
- 이를 위해 루트가 애그리거트의 도메인 기능을 구현한다.
- 외부 제공 기능은 루트의 public 메소드로만 연다. 내부 객체에도 메소드는 있지만, 그건 루트가 호출하기 위한 것이고 외부에서 직접 호출해 상태를 바꾸는 경로는 두지 않는다.
- 외부에서 애그리거트 내부 객체를 직접 변경하면 안 된다.

일관성을 지키는 두 습관
- setter를 public으로 만들지 않는다.
- 밸류 타입은 불변으로 구현한다.

그래서 따라오는 규칙
- 루트 속 밸류를 변경하려면, 루트가 제공하는 메소드에 새 밸류 객체를 전달하는 방법뿐이다.
- 루트가 내부 구현을 위해 내부 객체에 getter를 둘 수는 있다. 단 외부에서 상태를 바꿔 일관성이 깨지지 않도록 내부 객체를 불변으로 만든다.

## 3. 트랜잭션 경계

애그리거트는 일관성 단위다
- 각 애그리거트는 자기 상태와 불변식만 책임진다. 한 애그리거트가 다른 애그리거트의 내부 상태를 직접 바꾸면 안 된다.

여러 애그리거트를 변경해야 하는 경우
- 현재는 MSA가 아니고 컨텍스트별로 DB가 나뉘어 있지 않다. 단일 DB를 쓰는 지금은 한 트랜잭션에서 여러 애그리거트를 함께 커밋한다. 따라서 트랜잭션 경계는 응용 서비스가 설정한다. 
- 단, 이는 *트랜잭션을 묶는 책임*이 응용 서비스라는 뜻이지, *변경 로직 자체*를 응용 서비스에 두라는 뜻이 아니다. 두 애그리거트에 걸친 상태 변경이 한 애그리거트에 넣기 애매한 도메인 로직이면 도메인 서비스로 표현하고, 응용 서비스는 그 도메인 서비스를 호출한다.
- 응용 서비스든 도메인 서비스든, 애그리거트를 바꿀 때는 애그리거트가 공개한 public 메소드로 변경해야지 내부 객체를 직접 꺼내거나 조회해서 변경하면 안 된다.

## 4. 리포지토리

- 리포지토리는 애그리거트 단위로 존재한다.
- 애그리거트는 개념적으로 하나이므로 전체를 영속해야 한다. (Order 안에 OrderLine, Orderer가 있으면 모두 저장)
- 리포지토리는 완전한 애그리거트를 조회해서 제공해야 한다.

## 5. 애그리거트 간 참조와 조회

- 애그리거트끼리는 필드로 직접 참조하지 않고 ID로 참조한다.
- 여러 애그리거트의 연관 데이터를 모아 **화면에 보여주는 등 표시·조회 전용** 기능은 DAO로 필요한 데이터(여러 테이블을 조합한 read model)만 로딩하는 것을 고려할 수 있다.
- 단 이는 **도메인 규칙을 실행하지 않고 데이터만 조립**하는 경우에 한한다.
  - 도메인 규칙을 실행해 판정·계산하는 일은 상태를 바꾸지 않아도 **도메인 연산**이므로, 리포지토리로 애그리거트를 가져와 **루트의 행위로** 수행한다. 
  - 상태를 바꾸는 명령 흐름은 당연히 리포지토리로 완전한 애그리거트를 가져와 루트를 통해 변경한다.
- 조회용 DAO는 미리 만들지 않는다. ID 참조로 끊으면서 조회 성능 문제가 실제로 드러날 때 도입한다.

## 6. 팩토리

- 애그리거트가 가진 데이터로 다른 애그리거트를 생성해야 한다면, 애그리거트에 팩토리 메소드를 둔다.
- 생성에 알아야 할 정보가 많다면 직접 생성하지 않고 팩토리 클래스에 위임할 수 있다.

## 7. 엔티티와 밸류 구분

- 고유 식별자를 갖는지로 구분한다.
- ORM 매핑 테이블의 PK를 객체의 식별자로 착각하면 안 된다. PK는 RDB 저장을 위한 것이지, 저장 값이 고유 식별된다는 뜻은 아니다.

## 8. 도메인 서비스

- 상태 없이 로직만 구현한다. 필요한 상태는 다른 방법으로 전달받는다.
- 도메인 서비스 객체를 애그리거트에 주입하지 않는다.
- 이름은 반드시 도메인 관점으로 짓는다.

여러 애그리거트가 협력하거나 한 애그리거트에 넣기 애매한 로직
- 보통 복잡한 계산 로직이 여기 해당한다.
- 계산뿐 아니라 여러 애그리거트의 상태를 함께 변경해야 하는데 어느 한쪽에 넣기 애매한 로직도 도메인 서비스다.
  - 예: 계좌 이체. 한 애그리거트(출금 계좌)는 금액을 출금하고 다른 애그리거트(입금 계좌)는 입금한다. 두 애그리거트에 걸쳐 있어 한쪽에 둘 수 없으므로 도메인 서비스로 표현한다.
- 협력하는 애그리거트가 **같은 바운디드 컨텍스트**에 있으면, 그 애그리거트들을 **매개변수로 넘겨** 도메인 서비스 안에서 함께 변경하고 불변식을 검증한다. 한 애그리거트가 다른 애그리거트를 직접 바꾸면 안 되므로(3번) 이 조율을 도메인 서비스가 맡는다.
  - 여기서 '여러/둘 이상 애그리거트'는 애그리거트의 **종류(타입)가 아니라 개수**를 말한다. 같은 타입의 인스턴스 둘이 협력해도 다중 애그리거트 협력이다.
  - 예: 거래소 간 송금. 출발·도착 지갑(둘 다 wallet 컨텍스트, 같은 타입의 지갑 애그리거트 두 개)의 잔고를 매개변수로 받아 도메인 서비스로 구현한다.
  - 협력형인지 가르는 기준: **매개변수가 애그리거트 인스턴스**여야 하고, 상태 변경이 그 **애그리거트의 public 메소드 호출로** 나타나야 한다. 애그리거트 대신 VO 나 ID·값만 UseCase 로 전달만 하는 것은 협력형 도메인 서비스가 아니다.
- 도메인 서비스인지 응용 서비스인지 헷갈릴 때는, 그 로직이 애그리거트의 상태를 바꾸거나 값을 계산하는지 보면 된다. 그런 로직인데 한 애그리거트에 넣기 적합하지 않으면 도메인 서비스다.

외부 시스템을 사용해야 하는 도메인 로직
- 외부 시스템이나 타 도메인과의 연동도 도메인 서비스로 표현할 수 있다. 도메인 입장에서 연동은 단순 HTTP API 호출이 아니라 필요한 도메인 로직을 실행하는 것이기 때문이다.
- 협력해야 할 애그리거트가 **다른 바운디드 컨텍스트**에 있으면, 그 애그리거트를 가져와 직접 바꾸면 안 된다(3·9번). 상대 컨텍스트가 공개한 **UseCase에 위임**하는 연동형 도메인 서비스로 표현한다.
  - 예: `PlaceOrderService`에서 시장가 주문은 체결 후 잔고를 반영해야 한다. 주문은 trading, 잔고는 wallet 컨텍스트라 wallet 애그리거트를 직접 바꿀 수 없다. 그래서 주문 애그리거트가 "무엇을 반영할지"를 결정하고, 그 반영은 wallet 의 UseCase에 위임하는 연동형 도메인 서비스가 수행한다.
- 인터페이스는 **도메인 로직 관점**으로 짓는다. 이름이 도메인 의도를 드러내게 하고, 연동 대상·프로토콜 같은 구현 세부사항은 이름에 담지 않는다.
  - 예: 권한 시스템과 연동해 설문 생성 권한을 확인한다면 `RoleSystemClient.getRoles(userId)` 가 아니라 `SurveyPermissionChecker.hasCreationPermission(userId)` 처럼 짓는다.
  - 예: 추천 시스템과 연동해 추천 상품을 조회한다면 `RecSystemClient.getRecs(itemId)` 가 아니라 `ProductRecommendationService.getRecommendationsOf(productId)` 처럼 짓는다.
- 인터페이스는 도메인 패키지에, 구현체는 인프라 패키지에 둔다.
- 구현체는 외부 시스템 연동과 외부 모델 ↔ 도메인 모델 변환을 책임진다. 즉 ACL로도 작용한다.
  - 모델 변환은 구현체의 private 메소드로 해도 되고, 복잡하면 Translator 같은 별도 클래스로 분리해도 된다.
- 응용 서비스는 이 도메인 서비스를 호출해 외부 연동을 마치 도메인 로직처럼 사용한다. 응용 서비스 코드에는 연동 대상·프로토콜이 드러나지 않는다.

## 9. 바운디드 컨텍스트 간 관계

안티 코럽션 레이어
- 외부 도메인 모델이 내 도메인 모델에 영향을 주지 않도록 막는 완충 지대.
- 8번의 외부 연동형 도메인 서비스 구현이 이 역할을 한다.

공유 커널
- 두 바운디드 컨텍스트가 같은 모델을 공유해 중복 개발을 줄이는 방식.
- 한 팀이 모델을 임의로 변경하면 다른 팀이 깨질 수 있어, 변경에 두 팀의 합의가 필요하다.
- 그래서 커널은 오히려 개발을 지연시킬 수 있다.
