# 공통 멱등성 메커니즘

멱등성이 필요한 커맨드가 공유하는 처리 방식을 정의한다. 개별 기능은 이 공통 테이블·흐름을 재사용하고, 기능별 세부(재요청 응답 값, 도메인 락 등)는 각 기능 문서에서 다룬다.

## 방식 — 선점(gate-first) + 공통 멱등성 테이블 재사용

커맨드 트랜잭션에 진입하면 가장 먼저 공통 멱등성 테이블에 요청 키를 선점 기록하고, 이 테이블의 키 UNIQUE 제약 **하나로 동시성 제어와 중복 방지를 모두 처리**한다. 각 커맨드의 이력 테이블에는 멱등성 컬럼을 두지 않는다 — 키의 단일 소스는 멱등성 테이블이다.

즉 리소스 스코프의 `findBy...AndIdempotencyKey`(check-first, 이력 테이블에 키 보관) 방식은 쓰지 않는다. 키는 개별 리소스가 아니라 전역에서 유일하다.

## 키 저장소 — 공통 테이블

여러 커맨드가 공유하는 재사용 테이블 `idempotency_key` 로 둔다 (리소스 타입만 다르다). `idempotency_key` 컬럼에 UNIQUE 제약을 건다. 이 테이블·엔티티·리포지토리는 최초로 멱등성이 필요한 기능이 생성하고, 이후 기능은 리소스 타입만 추가해 재사용한다.

| 컬럼 | 설명 |
|------|------|
| 멱등성 키 | 클라이언트 생성 키. UNIQUE |
| 리소스 타입 | 커맨드 종류 (예: `EMERGENCY_FUNDING`). 다른 커맨드가 같은 테이블 공유 |
| 리소스 ID | 생성된 리소스 식별자. 선점 시 비어 있고, 이력 저장 후 채운다 |
| 생성 시각 | |

## 처리 흐름 (선점 → 처리 → 연결)

1. **선점**: 커맨드 트랜잭션의 첫 단계로 멱등성 테이블에 `(키, 리소스 타입, 리소스 ID=비움)` 을 기록한다. 이 기록은 지연 없이 즉시 DB 에 반영되어야 한다 — 그래야 키 UNIQUE 인덱스가 트랜잭션 상단에서 슬롯을 잠가, 뒤따르는 도메인 조회·검증·상태 변경보다 앞선다.
2. **처리**: 신규 키면 통과해 검증·상태 변경·이력 저장을 수행한다.
3. **연결**: 이력이 저장돼 식별자가 생기면 같은 트랜잭션에서 선점 행의 리소스 ID 를 그 이력 ID 로 채운다(감사·공통 테이블 일관성용 — 중복 응답이 이 값을 다시 읽지는 않는다).
4. **커밋**: 선점 행과 이력이 원자적으로 함께 커밋된다.
5. **중복**: 이미 쓰인 키면 선점 단계에서 중복으로 판정되어 처리를 중단하고, 아무것도 변경되기 전에 트랜잭션을 롤백한다.

## 중복 응답 처리 구조

응답 처리 구조는 모든 커맨드가 공유한다. **응답 DTO 에 어떤 값을 채우는지만 각 기능이 정의**하고, 아래 흐름은 그대로 재사용한다.

- **컨트롤러에서 최초 처리와 중복 요청을 try-catch 로 구분한다.** 애플리케이션 서비스는 "중복 요청"(`DuplicateRequestException`) 신호를 잡지 않고 그대로 던진다 — 서비스에는 제어흐름용 try-catch 를 두지 않는다. 컨트롤러가 정상 반환은 그대로 응답하되, 중복 예외가 올라오면 catch 해 재요청 응답을 만든다.
- DB 가 던지는 중복 키 예외는 영속성 어댑터에서 `DuplicateRequestException` 으로 번역한다. 그래서 서비스·컨트롤러는 DB 예외를 직접 보지 않고, 컨트롤러가 catch 하는 것도 이 예외다.
- **재요청은 상태를 바꾸지 않는다**(선점 단계에서 걸려 트랜잭션이 롤백). catch 후 컨트롤러는 **현재 상태를 읽기 전용으로 조회**(별도 조회 UseCase 호출)해 최초 처리와 **동일한 형태의 응답 DTO** 를 재구성한다. 무엇을 조회해 어떤 값을 채우는지는 각 기능 문서에서 정의한다.
- **최초 처리와 재요청 모두 같은 성공 상태 코드로 응답한다.** 사용자(그리고 클라이언트)는 멱등 중복 처리 여부를 알 필요가 없다.
