## 엔드포인트는 provider 무관 단일 경로

- 로그인에서는 소셜 프로바이더별로 엔드포인트를 나눴다. 이는 잘못된 설계다. 또한 로그아웃은 우리 서버 세션만 지우는 동작이라 어느 소셜로 로그인했는지와 무관하다.
- 따라서 `/api/auth/kakao/logout` 처럼 provider별로 나누지 않고 `POST /api/auth/logout` 하나만 둔다.

## 로그아웃은 인증 인터셉터를 거치지 않는다

- 로그아웃은 userId 가 필요 없다. "이 쿠키에 담긴 세션을 지워라"가 전부다. 특정 사용자 데이터를 조회하거나 건드리지 않으므로 인터셉터가 알려주는 userId 가 쓸모없다.
- `AuthInterceptor` 인터셉터로 막으면 멱등하게 처리하지 못한다. 세션이 방금 만료됐거나 다른 탭에서 이미 로그아웃한 사용자는 쿠키가 죽어 있다. 인터셉터를 거치면 이들이 로그아웃 요청을 할 때 401로 실패한다.
- 따라서 이미 로그아웃에 성공했는데 서버가 오류 응답을 보내므로 멱등하지 못하다.
- 따라서 로그아웃 컨트롤러가 요청 쿠키의 세션 ID를 직접 읽어, 있으면 지우고 없으면 그냥 성공 응답한다.

## provider 로그아웃/토큰 폐기 API 는 호출하지 않는다

- 카카오 로그아웃 API 는 "카카오가 우리에게 발급한 access token 을 폐기"하는 동작이다. 우리는 로그인 시 그 토큰을 저장하지 않고 회원번호 조회에 일회성으로 쓰고 버리므로, 폐기할 토큰 자체가 없다.
- 따라서 로그아웃 시 provider 를 향한 어떤 호출도 하지 않는다.

## API 명세

### `POST /api/auth/logout`

- Request Body 없음. 인증은 요청에 딸려오는 `SESSION` 쿠키로만 판단한다.
- 성공: `200` + `Set-Cookie: SESSION=; Max-Age=0; ...`(만료) + 아래 바디

```json
{
  "status": 200,
  "code": "SUCCESS",
  "message": "로그아웃되었습니다.",
  "data": null
}
```

- 세션 쿠키가 없거나 이미 만료된 경우에도 동일하게 `200` 을 반환한다. (멱등)

## 에러 응답

- 멱등 처리로 인해 로그아웃은 사실상 항상 `200` 이다. 세션 부재를 오류로 취급하지 않는다.
