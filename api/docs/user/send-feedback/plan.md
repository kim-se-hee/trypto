## 멱등성을 두지 않는다

멱등성을 이 기능에는 적용하지 않는다.
두 번 반영돼도 상관이 없다. 중복된 긴급 충전은 잔여 횟수와 잔고를 잘못되게 한다. 피드백의 재요청은 같은 글이 두 행 쌓일 뿐이며, 운영자가 한 건 지우면 끝난다.

## API 명세

`POST /api/feedbacks`

인증된 세션의 주인이 곧 작성자다. 작성자를 요청으로 받지 않는다.
경로를 `/api/users/me/feedbacks` 가 아니라 `/api/feedbacks` 로 두는 이유는, 피드백이 회원의 하위 자원이 아니라 독립 애그리거트이고 이번 범위에 회원별 조회가 없어 회원을 경로에 드러낼 필요가 없기 때문이다.

### Request Body

| 필드 | 타입 | 필수 | 검증 | 설명 |
|------|------|------|------|------|
| content | String | O | `@NotBlank` | 피드백 본문 |

```json
{
  "content": "송금 화면에서 수수료가 언제 빠지는지 알기 어려웠습니다. 미리 보여주면 좋겠습니다."
}
```

### Response

`201 Created`. 새 피드백 행이 실제로 생성되며, 멱등 재요청 같은 "아무것도 생성되지 않는 성공" 경로가 없으므로 201 이 정확하다.

응답 본문에 피드백 ID 를 담지 않는다. 스펙상 회원은 접수 사실만 알면 되고, 조회·수정·삭제가 없어 클라이언트가 ID 로 할 수 있는 일이 없다.

```json
{
  "status": 201,
  "code": "SUCCESS",
  "message": "피드백이 접수되었습니다.",
  "data": null
}
```

### 에러 응답

| code | status | 설명 |
|------|--------|------|
| UNAUTHENTICATED | 401 | 세션이 없거나 유효하지 않음 |
| INVALID_FEEDBACK_LENGTH | 400 | 본문이 20자 미만이거나 1000자 초과 (신규 코드) |
