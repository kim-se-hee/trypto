계획은 스펙의 비즈니스 규칙을 어떻게 구현할지 코드의 구조와 설계에 집중하는 단계이다 (how).
메서드 시그니처, 의사코드, 코드는 적지 않는다. 계획 문서 검수가 곧 코드 리뷰가 되면 의미가 사라진다.

계획은 사람과 대화하면서 만든다. AI 가 모르는 것은 추측해서 박지 말고 적극적으로 `AskUserQuestion` 으로 그 자리에서 묻는다.
당신의 사용자는 틀린 이야기를 할 때도 많으며 틀린 이야기를 합리적으로 지적당하는 것을 수상할 정도로 기뻐한다. 언제나 사용자의 주장을 비판적으로 사고해라.

## 입력

frontmatter `arguments: [scope, feature]` 로 명명된 위치 인자를 받는다.

- `$scope` = 기능이 속한 위치
    - api 모듈 (DDD 적용): `api/<context>` (예: `api/trading`)
    - 그 외 모듈: 모듈명 그대로 (예: `engine`, `collector`, `frontend`)
- `$feature` = 기능 이름 (kebab-case)

예시:
```
/plan api/trading place-order
/plan engine matching
```

인자가 부족하거나 형식이 맞지 않으면 사용자에게 호출 형태를 안내하고 종료한다.

## 사전 제약

기능 디렉터리에 `spec.md` 가 존재해야 한다. 없으면 사용자에게 알리고 종료한다 — 계획은 스펙을 구현 대상으로 삼는다.

## 디렉터리 구조

`docs` 는 항상 모듈 루트 바로 아래에 있다. api 모듈만 그 안에서 바운디드 컨텍스트로 한 단계 더 나뉜다.

- api (scope `api/<context>`):
  ```
  api/docs/<context>/<feature>/
  ├── index.md         ← 기능 메타 (spec 단계에서 이미 생성)
  ├── spec.md          ← what/why
  └── plan.md          ← how (이 스킬의 산출물)
  ```
- 그 외 (scope `<module>`):
  ```
  <module>/docs/<feature>/
  ├── index.md
  ├── spec.md
  └── plan.md          ← how (이 스킬의 산출물)
  ```

이하 본문에서 "기능 디렉터리" 는 위 구조의 `<feature>/` 디렉터리를 가리킨다.

## 작업 흐름

1. **파일 준비** 기능 디렉터리에 `plan.md` 가 없으면 빈 파일을 만든다.
2. **리서치** 아래를 읽어 어떤 것들을 구현할지 생각한다.
    - `spec.md` : 구현해야 하는 비지니스 규칙을 이해한다.
    - 의존 컨텍스트의 `dependency.md` : 이 기능을 구현하려면 먼저 구현돼야 할 타 컨텍스트의 UseCase 가 있을지 생각한다.
    - `docs/ddd-guideline.md` 와 이번 기능에 참여하는 컨텍스트들의 `aggregate.md` : 기존 애그리거트 중 재사용할 것과 새로 만들 것을 구분한다.
3. **작성** 사용자의 프롬프트에 따라 plan.md를 작성하거나 수정한다. 사용자의 의견을 물어야 한다면 반드시 `AskUserQuestion` 으로 그 자리에서 묻는다.
4. **충돌 검사** 사용자가 충돌 검사를 요청하면 plan 에 적힌 항목이 기존 설계와 어긋나지 않는지 확인한다. 리서치 리스트는 아래와 같다.
    - **도메인 모델 중복·책임 충돌** — api 모듈의 기능을 개발하는 경우 이번 plan 이 신규로 추가하려는 도메인 모델과 같은 개념의 도메인 모델이 같은 컨텍스트에 이미 존재하는지, 또는 같은 모델에 여러 plan 이 양립 불가능한 책임을 부여하고 있는지 확인한다.
    - **컨텍스트 경계 침범** — api 모듈의 기능을 개발하는 경우 다른 컨텍스트가 책임지는 일을 잘못된 컨텍스트에 두었는지 확인하다.
    - **컨택스트 맵 확인** — 의존 컨텍스트의 `dependency.md` 를 보고 호출하기로 한 UseCase·DTO 가 실제 노출돼 있고 시그니처가 맞는지 확인한다.
    - **이벤트·메시지 채널** — `docs/contracts/`를 보고 새 채널 이름이 기존과 겹치지 않는지, 페이로드 필드 의미가 기존 계약과 어긋나지 않는지 확인한다.
    - **스키마** — `docs/schema.md`를 보고 새 엔티티·필드명이 기존과 겹치거나 의미가 다른지 확인한다.

   충돌이 있으면 사용자에게 보고한다. 임의로 고치지 않는다.

## 양식

### `<feature>/plan.md`

```markdown
## 도메인 모델

### <모델 A> (신규 | 수정 | 신규 VO)
- 담당 비지니스 규칙 1
- 담당 비지니스 규칙 2

### <모델 B> (...)
- 담당 비지니스 규칙 1
- 담당 비지니스 규칙 2

## 타 컨텍스트 의존성

- <대상>.<UseCase> — <용도>
- <대상>.<UseCase> [신규 필요] — <왜 필요한지>

## 시퀀스 플로우


## task 목록

- [ ] <작업>
- [ ] <작업>
- [ ] <작업>

## API 명세        ← 외부 노출 시에만

## 이벤트 컨트랙트  ← 메시지 큐 사용 시에만

```
위 양식은 초안 작성을 위해 일반적인 경우 많이 포함되는 공통 섹션을 표현한 것이다. 경우에 따라서 얼마든지 새로운 요소가 들어갈 수 있다.
