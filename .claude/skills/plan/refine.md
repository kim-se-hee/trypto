기존 계획에 사용자 피드백을 반영한다. 피드백은 자유 텍스트로 들어오며, 도메인 모델 책임 재분배, 의존성 추가/제거, task 재구성 등이 일어날 수 있다.

피드백이 다루지 않은 부분은 임의로 수정하지 않는다. task 재정렬·문장 다듬기·관련 없는 모델 이동 등 사용자가 명시적으로 요구하지 않은 변경은 금지한다.

## 흐름

### 1. 대상 확인

`docs/<prefix>/<feature>/plan.md` 가 존재해야 한다. 없으면 사용자에게 알리고 종료한다.

### 2. 피드백 수신 / 변경 대상 식별

사용자가 자유 텍스트로 제공한 피드백을 받는다. 피드백 항목이 가리키는 섹션·모델·BR 번호를 결정한다. 
사용자가 명시적으로 지목하지 않은 경우 의미가 가까운 항목을 매칭한다. 매칭이 모호하면 사용자에게 확인을 요청한다.

### 3. 적용

피드백을 반영해 해당 섹션을 갱신한다.

### 4. 규칙 매핑 검증 표 재확인

피드백 반영 후 spec에 적힌 br이 모두 매핑되었는지 확인한다.

### 5. 기능 index.md 갱신

산출물 한 줄 요약이 변경되었으면 `docs/<prefix>/<feature>/index.md` 를 갱신한다.

### 6. 변경 보고

```
계획 갱신 완료: docs/<prefix>/<feature>/plan.md

변경 항목:
- <항목 요약>

미매핑 BR:
- BR<n>: <본문>
```

해당 항목이 없으면 그 섹션은 생략한다.
