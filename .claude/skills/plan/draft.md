신규 계획 초안을 작성한다. 이미 `<feature>/plan.md` 가 존재하면 사용자에게 알리고 종료한다 (덮어쓰지 않는다). `spec.md` 가 없어도 종료한다.

`SKILL.md` 의 양식·사전 리서치·BR 참조 정책을 그대로 따른다.

## 흐름

### 1. 사전 리서치

`SKILL.md` 의 사전 리서치 지침에 따라 트레일을 타고 자료를 탐색한다. 읽은 결과로 다음을 잡는다: 재사용 가능한 도메인 모델 후보, 가져다 쓸 수 있는 외부 인터페이스, 신규로 만들 항목 후보.

### 2. spec.md 본문 인용

`docs/<prefix>/<feature>/spec.md` 의 목적과 BR 목록을 메모한다. 이후 BR 매핑에 이용된다.

### 3. 도메인 모델 섹션 작성

각 모델마다 신규/수정 표시와 함께, 이번 기능에서 책임지는 BR 번호를 짧은 본문과 함께 적는다.

```markdown
### Order (수정)
- [3] 잔고 기반 주문 가능 검증
- [7] 출고 전 취소

### BalanceSnapshot (신규 VO)
- [5] 시점 잔고 + 미체결 주문 합 묶기
```

1단계 리서치에서 발견한 재사용 가능한 모델은 신규로 다시 만들지 않는다.

### 4. 크로스 컨텍스트 의존성

다른 컨텍스트·모듈에서 가져다 쓸 항목을 적는다. 인터페이스에 이미 존재하면 그대로, 새로 정의가 필요하면 `[신규 필요]` 마커를 붙이고 사유를 적는다.

### 5. 시퀀스 플로우

호출 흐름을 mermaid 다이어그램으로 그린다. 행위자(사용자, 컨트롤러, UseCase, 외부 시스템) 단위로 구분한다.

### 6. task 목록

위 결정들을 만드는 데 필요한 작업을 순서대로 체크박스(`- [ ]`)로 적는다. 위에서 아래로 순차 실행되므로 의존 순서를 그대로 반영한다.

```markdown
- [ ] 잔고 검증 도메인 모델 정의
- [ ] OrderController 추가
```

implement 단계에서 완료된 항목을 `- [x]` 로 갱신한다.

### 7. (조건부) API 명세

외부 노출 API 가 있는 기능이라면 엔드포인트와 요청/응답 모양을 적는다. 모듈의 API 컨벤션을 따른다.

### 8. (조건부) 이벤트 컨트랙트

메시지 큐를 사용하면 채널·페이로드를 적는다. `docs/contracts/` 의 기존 채널을 따르거나 신규 정의를 적는다.

### 9. 규칙 매핑 검증

spec 의 모든 BR 을 표로 나열한다. 각 BR 이 어느 도메인 모델 책임인지 적는다.

매핑되지 않은 BR 이 있으면 두 가지 처리 중 하나:
- 도메인 모델 섹션으로 돌아가 보강한다.
- "이번 작업 범위 외 — <이유>" 로 명시한다.

조용히 넘기지 않는다.

### 10. 기능 index.md 갱신

`docs/<prefix>/<feature>/index.md` 를 갱신한다.

- 단계: `plan` 으로 변경
- 산출물 목록에 `plan.md` 한 줄 추가

### 11. 보고

```
계획 작성 완료: docs/<prefix>/<feature>/plan.md

신규 도메인 모델: <목록>
수정 대상 모델: <목록>
신규 의존 인터페이스: <목록>
이번 범위 외 BR: <BR번호 + 이유>
```

해당 항목이 없으면 그 줄은 생략한다.
