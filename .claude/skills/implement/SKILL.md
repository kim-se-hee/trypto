description: >
  계획서를 코드로 옮기고 다중 관점 리뷰까지 통과시키는 워크플로우.
  /implement <scope> <feature> 로 호출하면 기능별 워크트리·브랜치를 만들고,
  task-implementer 가 plan.md 를 읽어 기능을 구현한 뒤 인수 테스트로 검증하고,
  5개 전문 리뷰어로 차단 이슈를 잡아 적용한다.
arguments: [scope, feature]
---

`/implement` 는 기능 디렉터리의 `plan.md` 한 장을 "merge 직전 상태" 까지의 완성도로 구현하기 위한 오케스트레이터다.
메인 세션은 디스패치·파일 갱신만 담당하고, 실제 코드 작성·리뷰는 모두 서브에이전트가 한다.

# 입력

- `$scope` — 기능이 속한 위치
  - api 모듈 : `api/<context>` (예: `api/trading`)
  - 그 외 모듈: 모듈명 그대로 (예: `engine`, `collector`, `frontend`)
- `$feature` — 기능 이름 (kebab-case, 예: `place-order`)

예시:
```
/implement api/trading place-order
/implement engine matching
```

인자가 부족하거나 형식이 맞지 않으면 사용자에게 호출 형태를 안내하고 종료한다.

# 디렉터리 구조

`docs` 는 항상 모듈 루트 바로 아래에 있다. api 모듈만 그 안에서 바운디드 컨텍스트로 한 단계 더 나뉜다.

- api (scope `api/<context>`): `api/docs/<context>/<feature>/`
- 그 외 (scope `<module>`): `<module>/docs/<feature>/`

이하 본문에서 "기능 디렉터리" 는 위 구조의 `<feature>/` 디렉터리를 가리킨다.

# 사전 제약

- 기능 디렉터리에 `plan.md` 가 존재해야 한다. 없으면 종료한다.
- 이미 워크트리·브랜치가 있으면 종료한다 (재진입은 후속 작업).

# 흐름

흐름은 **구현 후 리뷰 리팩토링 루프**를 수행하는 것이다.
구현은 한 번, 리뷰는 블로킹 이슈가 0건이 될 때까지 반복하는 루프다.
구현 단계가 실패하면 리뷰는 하지 않고 스킬을 종료한다.

## 작업 위임 방식

메인은 코드를 짜지 않는다. 코드 작성은 전부 `task-implementer` 서브에이전트가 하며, 워커는 **필요한 문서를 직접 읽어** 작업한다. 메인은 그 문서를 만들고 큐레이션하며(plan.md 는 사전에 사용자가, review.md 는 메인이 6단계에서 작성), 상태(index.md 단계·review.md 체크박스)를 관리한다.

- **구현 단계**: 워커에게 "plan.md 를 읽어 구현하라"고 지시한다.
- **적용 단계**: 워커에게 "review.md 의 차단 이슈를 반영하라"고 지시한다.

## 구현 단계

### 0. plan.md 확인

기능 디렉터리에 `plan.md` 가 있는지 확인한다. 없으면 종료. 구현 단계는 워커가 plan.md 를 통째로 읽으므로 메인은 여기서 task 를 추출하지 않는다. (재진입은 사전 제약의 워크트리·브랜치 존재 체크로 걸러진다.)

### 1. 워크트리 + 브랜치 생성

명명 규칙:
- 워크트리 경로: `../trypto-<feature>`
- 브랜치명: `feat/<feature>`

```bash
git worktree add -b feat/<feature> ../trypto-<feature>
```

이미 워크트리·브랜치가 존재하면 사용자에게 알리고 종료.

### 2. 구현 위임

`task-implementer` 서브에이전트에 **한 번** 위임한다. 프롬프트에 `scope`, `feature`, 그리고 **"구현 모드: plan.md 를 읽어 이 기능을 구현하라"** 지시를 전달한다.

정상 종료 후 워커가 만든 커밋 목록과 완료 보고를 받는다. 워커가 막혀 보고하면 그 메시지를 사용자에게 그대로 전달하고 즉시 종료한다.

### 3. 인수 테스트 실행

**scope 가 `api/*` 가 아니면 이 단계 건너뛰고 4단계로.** 인수 테스트는 api 모듈의 기능에만 존재한다.

scope 가 `api/*` 이면 구현이 끝난 뒤 `acceptance-test-runner` 서브에이전트에 위임한다. 인수 테스트(`.feature`/step)는 사용자가 직접 작성해 둔 것을 그대로 실행한다.
프롬프트에 `scope`, `feature` 만 전달한다.

- **통과**: 4단계로.
- **실패**: 러너가 보고한 메시지를 사용자에게 그대로 전달하고 종료

### 4. 구현 단계 index.md 갱신

`index.md` 의 `단계` 줄을 `단계: implement` 로 갱신한다.


## 리뷰 단계

### 5. 리뷰 (병렬)

이번 회차의 **리뷰 범위**를 메인이 SHA 범위(diff range)로 정해 리뷰어에 넘긴다. 범위 산정 주체는 메인이고, 리뷰어는 직접 범위를 정하지 않는다.

- **1차 리뷰:** `main...HEAD` — 브랜치 생성 후 전체 변경.
- **2차 이후 재리뷰:** `<직전 회차 PRE_FIX>..HEAD` — 직전 회차 적용 루프가 바꾼 변경분만. (`PRE_FIX` 는 7단계에서 기록한다.) 즉 회차가 갈수록 범위는 "방금 고쳐진 변경분"으로 좁혀진다.

메인은 `ddd-reviewer` 호출 여부를 판단하기 위해서만 `git diff --name-only <범위>` 를 떠서 `api/` 경로 포함 여부를 확인한다. 이 목록은 메인 내부 판단용이며 리뷰어에 넘기지 않는다.

다음 리뷰어들을 **단일 메시지에서 병렬로** 호출한다.

- `ddd-reviewer` — 위 `--name-only` 결과에 `api/` 경로가 하나라도 포함된 경우에만 호출. 그 외엔 스킵.
- `oop-reviewer`
- `concurrency-reviewer`
- `performance-reviewer`
- `convention-reviewer` — scope 의 모듈에 `docs/conventions.md` 가 있으면 호출. 기준 문서가 없는 모듈이면 리뷰어가 스스로 스킵(승인)하므로 항상 호출해도 무방하다.

각 호출 프롬프트에 `scope`, `feature`, 그리고 **위에서 정한 리뷰 범위(SHA range)** 를 전달한다. 리뷰어는 그 범위로 직접 `git diff <범위>` 를 떠서 변경분을 보고 검토한다(직접 범위를 새로 정하지 않는다).

### 6. 이슈 리스트 작성

메인이 리뷰어 반환값을 종합해 `review.md` 에 체크박스 리스트로 저장한다. 양식:

```markdown
## N차 차단 이슈

- [ ] **[파일경로:라인] 이슈 제목** (출처: ddd | oop | 동시성 | 성능 | 컨벤션)
  - **설명:** ...
  - **수정 제안:** ...

- [ ] **[파일경로:라인] ...**
  - ...

## N차 참고 이슈 (수정 안 함, 보고용)

- [파일:라인] 제목 (출처)
- ...
```

분류 규칙:
- **차단 이슈**: 모든 리뷰어가 **차단**으로 표시한 것의 합집합(컨벤션 리뷰어의 위반은 전부 차단). (리뷰어는 diff가 추가·수정한 라인의 문제만 검토하므로, 차단 이슈는 전부 이번 변경이 유발한 것이다.)
- **참고 이슈**: 리뷰어가 **참고**로 표시한 것. 자동 적용 대상이 아님.

차단 이슈가 0건이면 8단계 통과 조건을 곧장 만족한 것으로 보고 9단계로. 1건 이상이면 7단계로.

회차마다 *기존 내용을 지우지 않고* 새 섹션(`## 2차 차단 이슈` 등)을 아래에 추가한다 — 사용자가 어떤 이슈가 나왔다 사라졌는지 한 파일에서 추적할 수 있게.

### 7. 적용

**적용을 시작하기 전에** 현재 HEAD SHA 를 `PRE_FIX_<현재회차>` 로 기록한다(`git rev-parse HEAD`). 이 값은 다음 회차 재리뷰(5단계)가 "이번 회차에 바뀐 파일만" 추려내는 기준점이다. 적용 커밋이 여러 개여도 `git diff PRE_FIX..HEAD` 가 그 사이 변경을 전부 포함하므로, 마커는 회차 *끝*이 아니라 적용 *시작 직전*이어야 한다.

`task-implementer` 서브에이전트에 **한 번** 위임한다. 프롬프트에 `scope`, `feature`, 그리고 **"적용 모드: review.md 의 미체크 차단 이슈를 반영하라"** 지시를 전달한다. 워커가 `review.md` 를 직접 읽어 미체크 `[ ]` 차단 이슈를 이슈당 한 커밋으로 고친다. 참고 이슈(체크박스 없는 항목)는 자동 적용 대상이 아니므로 워커가 건드리지 않는다.

1. 정상 종료 후, 워커가 **`완료`로 보고한 이슈**의 `review.md` 줄을 `- [ ]` → `- [x]` 로 갱신한다. (제목으로 매칭)
2. 워커가 막히면 (해석 불가, 수정 후에도 검증 실패 등) 그때까지 완료분만 체크 갱신한 뒤 그 메시지를 사용자에게 그대로 전달하고 중단한다.

### 8. 재리뷰

모든 차단 이슈가 체크됐으면 5단계를 다시 한다. 종료 조건:

- **최대 3회**: 리뷰는 최대 3회 실행. 3회째에도 새 차단 이슈가 나오면 사용자에게 잔여 이슈를 보고하고 종료.
- **0건이면 통과**: 새 회차에서 차단 이슈가 0건이면 9단계로.

### 9. 리뷰 단계 index.md 갱신

8단계가 "차단 이슈 0건 통과" 로 끝났을 때만 기능 디렉터리의 `index.md` 의 `단계` 줄을 `단계: review` 로 갱신한다.

## 최종 보고

```
완료: <기능 디렉터리>

[구현]
구현: 완료 / 막힘 (<사유>)
커밋: <개수>
인수 테스트: 통과 / 실패 (n/m) / 해당 없음 (api 모듈 아님)

[리뷰]
변경 파일: M개
리뷰 사항: <review.md 경로>
차단 이슈: 통과 (0건, n차) / 잔존 N건 (n차)
참고 이슈: N건 (수정하지 않음)
```
