신규 스펙 초안을 작성한다. 이미 `<feature>/spec.md` 가 존재하면 사용자에게 알리고 종료한다 (덮어쓰지 않는다).

`SKILL.md` 의 마커·번호 정책·BR 정의를 그대로 따른다.

## 흐름

### 1. 충돌 후보 탐색

`docs/<prefix>/index.md` 를 Read 하여 같은 prefix 의 기능 목록을 확인한다. 신규 기능과 의미가 겹칠 수 있는 후보를 추린다.

다른 모듈의 영향이 있다면 `docs/contracts/index.md` 도 함께 본다.
api 모듈의 스펙 초안을 작성한다면 다른 컨텍스트의 영향을 확인하기 위해 `docs/<prefix>/dependency.md` 를 함께 본다.

### 2. 후보 본문 인용

추린 후보 기능들의 `spec.md` 를 Read 한다. 신규 스펙 작성 시 참고할 정의·규칙을 메모한다.

### 3. 기능 폴더 초기화

```
docs/<prefix>/<feature>/
├── index.md
└── spec.md
```

`index.md` 초기 양식:

```markdown
- 단계: spec
- 리뷰 회차: 0/3
- QA 회차: 0/3

## 산출물
- [spec.md](spec.md) — <한 줄 요약>
```

`spec.md` 초기 양식:

```markdown
## 목적

## 비즈니스 규칙

## 추측 근거
```

### 4. 목적 작성

기능이 무엇이고 왜 필요한지 짧게 적는다. 구현 방법은 적지 않는다.

### 5. 비즈니스 규칙 작성

사용자가 제공한 정보 + 기존 스펙 인용 + 합리적 추론으로 BR 을 채운다. 1번부터 순차 부여한다.

비지니스 규칙에는 마커를 부착한다.
마커 부착 기준:
- 사용자가 답해야 진행 불가 → `[BLOCK]`
- 추론으로 채웠으나 확인 필요 → `[ASSUME]`
- 사용자 검토가 필요없는 자명한 항목 → 마커 없음

### 6. 추측 근거 섹션 작성

`[ASSUME]` 이 붙은 BR 마다 짧은 근거를 적는다. `[BLOCK]` 은 어떤 답이 필요한지를 적는다.

```markdown
## 추측 근거
- [3] 일반적인 거래소 관행으로 추정
- [5] 사용자 답변 필요: 부분 체결 허용 여부
```

### 7. prefix 인덱스 갱신

`docs/<prefix>/index.md` 의 기능 목록에 한 줄을 추가한다.

```markdown
- [<feature>](<feature>/) — <한 줄 요약>
```

`docs/<prefix>/index.md` 가 없으면 생성한다.

### 8. 미해결 마커 보고

작성한 스펙에서 `[BLOCK]` / `[ASSUME]` 이 붙은 BR 번호와 본문을 사용자에게 일람으로 보여준다. 사용자는 이를 보고 `/spec refine` 으로 답을 줄지 결정한다.

보고 형식:

```
스펙 작성 완료: docs/<prefix>/<feature>/spec.md

미해결 마커:
- [BLOCK] BR<n>: <본문>
- [ASSUME] BR<n>: <본문>
```

마커가 없으면 "미해결 마커 없음" 이라고 알린다.
